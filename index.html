🔴 INSIGHT 1: NRR Masking Silent Churn Crisis

The Paradox: NRR appears healthy at 124%, suggesting strong expansion-driven growth. However, 28.5% of accounts show zero logins for 60+ days before officially churning. This "silent churn" means we're over-investing in expansion while under-investing in resurrection. The 342 at-risk accounts represent $680K in ARR exposure. Historical pattern: 70% of silent churners had declining engagement velocity for 6 weeks before going dark.

Recommended Actions

  • Build "at-risk" playbook triggered by engagement drop (not cancellation notice)
  • Launch re-engagement campaign at day 45 of inactivity (test before day 60)
  • Implement account health score: login frequency + feature breadth + team size
  • Reallocate 20% of expansion budget to resurrection campaigns

The Constraint: Accounts using 4+ features generate 3x more expansion revenue than single-feature users. Yet 70% of users never discover feature #3 (Integrations/Automation). The problem isn't product value—it's feature discovery. Current onboarding focuses on first action (messaging) but fails to expose multi-feature value within the critical 7-day window. API Access and SSO (enterprise features) have lowest adoption but highest expansion correlation (3.5x).

Recommended Actions

  • Redesign onboarding to force multi-feature exposure in first 7 days
  • Measure TTV to Feature #3 (not just first action) as primary onboarding KPI
  • Build "feature discovery" prompts at day 3 and day 5 based on usage patterns
  • Create integration marketplace highlights for Team plan users

The Warning: Support ticket volume per $100K MRR increased 25% after major UI redesign (May 2026), but CSAT remained flat. This is critical—users are confused but not angry (yet). 53% of tickets are "confusion" type vs. 15% bugs and 32% feature requests. This is a leading indicator of future churn if unresolved. Historical data shows confusion ticket spikes precede churn by 90 days. The redesign improved aesthetics but degraded discoverability.

Recommended Actions

  • Categorize tickets by type: confusion vs bug vs feature request (target: confusion <25%)
  • If confusion >40%, halt new features and run usability sprints
  • Implement in-app contextual guidance for redesigned features
  • A/B test old vs new UI for power users—measure task completion time
